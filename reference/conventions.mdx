TFW conventions define every rule that governs how work is structured, named, tracked, and completed in a Trace-First Workflow project. This page is the authoritative reference — every artifact, status, budget, and role lock defined here is the single source of truth for a running project.

## Required artifacts

Every TFW project contains the following files at the project root and inside `.tfw/`:

| Artifact | Location | Purpose |
|---|---|---|
| `README.md` | Project root | Human explanation: why/what/how. Contains the Task Board. |
| `AGENTS.md` | Project root | AI agent behavior rules for the project. |
| `TECH_DEBT.md` | Project root | Accumulated tech debt: observations → triage → registry. |
| `KNOWLEDGE.md` | Project root _(optional)_ | Project knowledge index: architecture, decisions, legacy. Template: `.tfw/templates/KNOWLEDGE.md`. |
| `RELEASE.md` | Project root _(optional)_ | Project release strategy and context. Template: `.tfw/templates/RELEASE.md`. |
| `.tfw/README.md` | `.tfw/` | TFW philosophy, lifecycle, values. |
| `.tfw/conventions.md` | `.tfw/` | Project conventions (authoritative rules). |
| `.tfw/glossary.md` | `.tfw/` | Project glossary. |
| `.tfw/PROJECT_CONFIG.yaml` | `.tfw/` | Project configuration: task prefix, scope budgets, template/workflow paths, research limits, knowledge settings. |
| `.tfw/VERSION` | `.tfw/` | Current framework version (semver, single line). |
| `.tfw/CHANGELOG.md` | `.tfw/` | Version history in Keep a Changelog format. |
| `.tfw/compilable_contract.md` | `.tfw/` | Build-time compilation specification: Source Manifest, Reference Format, Output Structure. |

**Templates** (`.tfw/templates/`): `HL.md`, `TS.md`, `RF.md`, `ONB.md`, `RES.md`, `REVIEW.md`, `KNOWLEDGE.md`, `RELEASE.md`.

**Workflows** (`.tfw/workflows/`): `init.md`, `plan.md`, `research/base.md`, `handoff.md`, `review.md`, `resume.md`, `docs.md`, `knowledge.md`, `release.md`, `update.md`, `config.md`.

## Artifact types

TFW defines six primary artifact types. Each strictly follows its canonical template. See [artifact types](/concepts/artifact-types) for full conceptual detail.

| Type | Purpose |
|---|---|
| **HL** | Context and frame for a task — the "map of meaning". Not a task itself. |
| **RES** | Structured investigation artifact, produced via Briefing → Gather → Extract → Challenge stages. |
| **TS** | Task definition. Always self-contained: inputs, outputs, constraints, Definition of Done. |
| **RF** | Results, facts, final data. RF has priority as source of truth. Contains mandatory Observations table. |
| **ONB** | Executor's structured report before starting: understanding, blocking questions, risks, inconsistencies. |
| **REVIEW** | Formal coordinator report after reviewing RF: 9-point checklist, verdict, tech debt triage. |

## Task numbering and file naming

The task prefix is defined in `.tfw/PROJECT_CONFIG.yaml` (`tfw.task_prefix`). All artifact filenames must include the task ID (`{PREFIX}-{N}`) or a Phase identifier — a filename without either is an error.

| Artifact | Format | Example |
|---|---|---|
| Master HL | `HL-{PREFIX}-{N}__{title}.md` | `HL-PROJ-3__tfw-setup.md` |
| Single-phase RES | `RES__{PREFIX}-{N}__{title}.md` | `RES__PROJ-3__tfw-setup.md` |
| Single-phase TS | `TS__{PREFIX}-{N}__{title}.md` | `TS__PROJ-3__tfw-setup.md` |
| Single-phase RF | `RF__{PREFIX}-{N}__{title}.md` | `RF__PROJ-3__tfw-setup.md` |
| Single-phase ONB | `ONB__{PREFIX}-{N}__{title}.md` | `ONB__PROJ-3__tfw-setup.md` |
| Single-phase REVIEW | `REVIEW__{PREFIX}-{N}__{title}.md` | `REVIEW__PROJ-3__tfw-setup.md` |
| Phase RES | `RES__Phase{X}__{title}.md` | `RES__PhaseA__conventions.md` |
| Phase TS | `TS__Phase{X}__{title}.md` | `TS__PhaseA__conventions.md` |
| Phase RF | `RF__Phase{X}__{title}.md` | `RF__PhaseA__conventions.md` |
| Phase ONB | `ONB__Phase{X}__{title}.md` | `ONB__PhaseA__conventions.md` |
| Phase REVIEW | `REVIEW__Phase{X}__{title}.md` | `REVIEW__PhaseA__conventions.md` |

Task folder: `tasks/{PREFIX}-{N}__{title}/`

Research stage files (`briefing.md`, `gather.md`, `extract.md`, `challenge.md`) live in `tasks/{ID}/research/`. File existence equals stage completion. The final `RES__*.md` at the task root is the synthesis of all stages.

For multi-iteration research, each iteration gets its own subfolder and RES file: `research/` → `research2/` → `research3/`. Research folders accumulate — never delete or overwrite previous iterations. Each folder is a trace; deleting them means deleting reasoning.

## Task statuses

The full lifecycle flows as follows:

```
⬜ TODO → 📝 HL_DRAFT → 🔬 RES → 🟡 TS_DRAFT → 🟠 ONB → (develop) → 🟢 RF → 🔍 REV → 📚 KNW → ✅ DONE
                                                                            │
                                                                  ┌─────────┴─────────┐
                                                                  🔄 REVISE          ❌ REJECT
                                                               (back to dev)    (user decides)
```

| Status | Emoji | Meaning |
|---|---|---|
| TODO | ⬜ | Task planned, HL not started |
| HL_DRAFT | 📝 | HL being drafted, awaiting review/approval |
| RES | 🔬 | Research in progress (optional — user can skip to TS_DRAFT) |
| TS_DRAFT | 🟡 | TS written, awaiting approval for execution |
| ONB | 🟠 | Onboarding: executor studying the task |
| RF | 🟢 | Execution complete, RF written |
| REV | 🔍 | Review: reviewer checking RF |
| KNW | 📚 | Knowledge capture: tfw-docs + tfw-knowledge applied (optional — reviewer can pre-close with N/A) |
| DONE | ✅ | Task closed, traces updated |
| BLOCKED | ❌ | Blocked by dependency |

Review verdicts:
- **APPROVE** — all ok → KNW (run tfw-docs + tfw-knowledge), then DONE
- **REVISE** — specific issues → back to execution (same task)
- **REJECT** → user decides: (a) HL_DRAFT (rework HL), (b) RES (new research), (c) TS_DRAFT (rewrite TS)

## Scope budgets

Scope budgets are limits applied per phase to keep changes reviewable. Values are configured in `.tfw/PROJECT_CONFIG.yaml` under `tfw.scope_budgets`. When a budget is exceeded, split the phase.

| Parameter | Default | Config key | Rationale |
|---|---|---|---|
| Files per phase | 14 | `max_files_per_phase` | Agent maintains full context of changed files |
| New files per phase | 8 | `max_new_files` | Limits blast radius of new abstractions |
| LOC per phase | 1200 | `max_loc` | Keeps changes reviewable in one pass |
| Modified files | 12 | `max_modified_files` | Prevents scattered, hard-to-review diffs |

## Execution modes

TFW supports two execution modes. The mode applies to how the AI agent acts during a session.

| Mode | Default | Behavior |
|---|---|---|
| **CL (Chat Loop)** | Yes | AI proposes steps; human approves and executes. AI does not run external actions without approval. Used for planning, review, architecture decisions, and anything touching external systems. |
| **AG (Autonomous)** | No — explicit request only | AI works independently within the file system. Only for pre-approved scope (e.g., executing an approved TS). Must fail safely if context is missing. |

## Workflows

TFW defines 11 canonical workflows in `.tfw/workflows/`. Each workflow declares a Role Lock at the top.

| Workflow | Role | Purpose |
|---|---|---|
| `init.md` | Coordinator | Discover project → interview → knowledge → setup → verify |
| `plan.md` | Coordinator | Research → HL → RESEARCH gate → scope decision → TS |
| `research/base.md` | Researcher | Structured investigation → RES artifact (pipeline or standalone) |
| `handoff.md` | Executor | Context load → ONB → execute → RF |
| `review.md` | Reviewer | Read RF → checklist → verdict → tech debt → traces |
| `resume.md` | Coordinator | Locate task → status matrix → decide next phase |
| `docs.md` | Coordinator | Update KNOWLEDGE.md and TECH_DEBT.md after task completion |
| `knowledge.md` | Coordinator | Consolidate fact candidates into verified project knowledge (Orient → Gather → Consolidate → Prune) |
| `release.md` | Coordinator | Read RELEASE.md → scope release → version bump → CHANGELOG → tag |
| `update.md` | Coordinator | Fetch upstream → compare versions → categorize changes → update checklist → re-sync adapters |
| `config.md` | Coordinator | Interactive config change → propagate to all inline values |

## Tool adapter pattern

`.tfw/` is the tool-agnostic core — one copy per project. Each development tool reads its own entry point, which references `.tfw/`:

```
CLAUDE.md        → "Read .tfw/README.md, follow .tfw/conventions.md"
.cursor/rules    → "Read .tfw/README.md, follow .tfw/conventions.md"
.agent/rules     → "Read .tfw/README.md, follow .tfw/conventions.md"
```

Adapters are chosen at project initialization. See [adapter overview](/adapters/overview) for setup details for Claude Code, Cursor, and Antigravity.

## Context loading order

When starting a new session, agents must load context in this strict order:

<Steps>
  <Step title="AGENTS.md">
    Load the project's agent behavior rules first.
  </Step>
  <Step title="TFW core files">
    Load `.tfw/conventions.md` and `.tfw/glossary.md`.
  </Step>
  <Step title="KNOWLEDGE.md">
    Load `KNOWLEDGE.md` if it exists.
  </Step>
  <Step title="Task artifacts">
    Load the relevant HL, TS, and RF files for the current task.
  </Step>
</Steps>

## Quality standard

The quality standard for TFW work is non-negotiable:

- No placeholders. Results must be usable without manual edits.
- If a result is wrong, fix the prompt or context and retry until quality is met.
- Tasks are atomic and human-verifiable.
- **Content language:** Template structure (headings, labels, field names) is always English. Artifact content is filled in the language specified by `tfw.content_language` in `PROJECT_CONFIG.yaml` (default: `en`). The agent must check this value before writing artifacts.

### Design rules

| Rule | Description |
|---|---|
| **Token density** | Workflow instructions ≤ 1200 words. Templates own format; workflows reference templates. |
| **Inline enforcement** | Enforcement-critical values must be inline (Pattern A: defaults + config key). Pure references (Pattern B) are broken. |
| **DNA/Library** | Role Lock and Mindset are always inline. Reference data goes via ref-inside-step. Each step is self-contained; the ref adds precision. |
| **Progressive Disclosure** | Agent loads only what it needs now. Mode files are loaded at Step 2, not at start. |

## Anti-patterns

<Warning>
The following actions are explicitly prohibited in any TFW project. Violating these rules breaks the trace and degrades project quality.

- Executor starts coding before all blocking questions are resolved
- Executor skips reading HL and goes straight to code
- Coordinator skips review and closes without a REVIEW file
- RF file does not mention test results or observations
- TS is written without an approved HL
- Executor modifies Master HL without coordinator approval
- Executor makes architectural decisions not in HL
- Executor modifies files outside TS scope (even "obvious fixes")
- Executor does "bonus fixes" without documenting in RF deviations
- Executor writes RF before build/lint passes
- Executor sees tech debt or dead code but does not report it in Observations
- Coordinator ignores executor Observations — must triage to TECH_DEBT.md
- Coordinator writes ONB, RF, or implements code → **Role Lock violation**
- Executor writes HL, TS, or changes scope → **Role Lock violation**
- Executor writes REVIEW file → **Role Lock violation**
</Warning>

## Role lock

Each workflow declares a Role Lock at the top. The agent must refuse any action outside the locked role.

| Workflow | Role Lock | Permitted artifacts | Forbidden artifacts |
|---|---|---|---|
| `init.md` | Coordinator | RES, RF, project config files | HL, TS, code |
| `plan.md` | Coordinator | HL, TS | ONB, RF, RES, REVIEW, code |
| `research/base.md` | Researcher | RES, research/ stage files | HL, TS, ONB, RF, REVIEW, code |
| `handoff.md` | Executor | ONB, RF, code | HL, TS, RES, REVIEW |
| `review.md` | Reviewer | REVIEW | ONB, RF, HL, TS, code |
| `resume.md` | Coordinator | Status matrix, Phase HL, Phase TS | ONB, RF, RES, REVIEW, code |
| `docs.md` | Coordinator | KNOWLEDGE.md, TECH_DEBT.md | code |
| `release.md` | Coordinator | VERSION, CHANGELOG.md | code |
| `update.md` | Coordinator | `.tfw/` files, adapter copies | code |
| `config.md` | Coordinator | PROJECT_CONFIG.yaml, workflow files, convention files, adapter copies | code |

### Hard stop rule

- When a **Coordinator** reaches the end of planning (TS approved): inform the user, then instruct them to start `/tfw-handoff`. Do not continue into execution.
- When an **Executor** finishes RF: inform the user, then instruct them to start `/tfw-review`. Do not write a REVIEW file.
- When a **Researcher** finishes RES: inform the user, then instruct them to continue with `/tfw-plan`. Do not write HL or TS.
