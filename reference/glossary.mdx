This glossary defines every term used in Trace-First Workflow. Terms are organized by category. For naming rules, status flows, and Role Lock tables, see the [conventions reference](/reference/conventions).

## Execution modes

<AccordionGroup>
  <Accordion title="CL (Chat Loop Mode)">
    The default execution mode. The AI proposes steps; the user approves and executes external actions. The AI does not run external commands without approval. Used for task planning, code review, architecture decisions, and any work touching external systems.
  </Accordion>
  <Accordion title="AG (Autonomous Mode)">
    The AI works independently within the file system. Only used for pre-approved scope — for example, executing an approved TS. The agent must fail safely if context is missing. Requires explicit request to activate.
  </Accordion>
  <Accordion title="Read-only AG">
    A mode within the RESEARCH workflow where the agent autonomously reads project files and web sources but writes only to the RES artifact. No code changes, no other file modifications.
  </Accordion>
</AccordionGroup>

## Artifact types

Full definitions, naming rules, and format requirements are in [conventions.md §3](/reference/conventions) and [artifact types](/concepts/artifact-types).

<AccordionGroup>
  <Accordion title="HL (High Level)">
    Context and frame for a task — the "map of meaning". Not a task itself. Produced by the Coordinator during the plan workflow. Strictly follows `.tfw/templates/HL.md`. Must be approved before a TS is written.
  </Accordion>
  <Accordion title="RES (Research Report)">
    Structured investigation artifact for the RESEARCH stage. Living document: decisions at top, stage logs below. Produced via Briefing → Gather → Extract → Challenge stages in the `research/` subfolder. The RES file at task root is the synthesis (Decisions, Hypotheses, HL Recommendations, Conclusion). Strictly follows `.tfw/templates/RES.md`.
  </Accordion>
  <Accordion title="TS (Task Spec)">
    Task definition for a single phase. Self-contained: scope, steps, acceptance criteria, Definition of Done. Written by the Coordinator after HL is approved. Strictly follows `.tfw/templates/TS.md`.
  </Accordion>
  <Accordion title="RF (Result File)">
    Results, decisions, artifacts from execution. RF has priority as source of truth. Contains a mandatory Observations table (structured, typed). Written by the Executor after implementation. Strictly follows `.tfw/templates/RF.md`.
  </Accordion>
  <Accordion title="ONB (Onboarding Report)">
    Structured executor report written before starting work: understanding of the task, blocking questions, risks, inconsistencies. The Coordinator or human answers directly in the file (Q&A format). Strictly follows `.tfw/templates/ONB.md`.
  </Accordion>
  <Accordion title="REVIEW (Review Report)">
    Formal report written by the Reviewer after reading the RF. Includes a 9-point checklist, a verdict (APPROVE / REVISE / REJECT), and tech debt triage. Strictly follows `.tfw/templates/REVIEW.md`.
  </Accordion>
  <Accordion title="TECH_DEBT.md">
    Accumulated tech debt registry at the project root. Fed by: executor Observations in RF → coordinator triage in REVIEW → appended to TECH_DEBT.md.
  </Accordion>
  <Accordion title="KNOWLEDGE.md">
    Optional project knowledge index at the project root. Central map of architecture, decisions, legacy, and principles. Updated via the `tfw-docs` workflow. Principle: index, don't duplicate. Template: `.tfw/templates/KNOWLEDGE.md`.
  </Accordion>
  <Accordion title="RELEASE.md">
    Optional project-level artifact defining release strategy: audience, triggers, version scheme, release checklist. Template: `.tfw/templates/RELEASE.md`.
  </Accordion>
</AccordionGroup>

## Knowledge terms

<AccordionGroup>
  <Accordion title="Fact Candidate">
    A raw observation about the project recorded during work in an artifact's Fact Candidates section. Not a verified fact — it becomes a fact after `/tfw-knowledge` consolidation. Quality filter: "Would the next agent decide differently knowing this?" Fact Candidates appear in RF §6, RES, and REVIEW §5. Categories: environment, process, stakeholder, constraint, convention, domain, context, risk, philosophy.
  </Accordion>
  <Accordion title="Strategic Insight">
    Human-sourced domain knowledge captured with deep analytical synthesis. Appears in three contexts: HL §11 "Strategic Insights (Planning)", RF §7 "Strategic Insights (Execution)", RES "Strategic Insights (Research)". The agent's cognitive mode is to capture the insight and then add implications — what does it mean for the project? High-value signals include user corrections, emotional statements, vision framing, and alternative selection. Contrast with Fact Candidate (pure reporting, no interpretation).
  </Accordion>
  <Accordion title="Value Flow">
    A visual section in the HL template (§3.2). Visualizes how value gets created — the process from user pain through pipeline steps to value delivered. Cognitive mode: strategic/value-oriented (INPUT→PROCESSING→OUTCOME). Distinct from §3.1 Result Visualization (outcome preview).
  </Accordion>
  <Accordion title="Findings Map">
    A visual section in the RES template. Visualizes research findings: root cause analysis, hypothesis trees, priority matrices, relationship maps between discoveries. Cognitive mode: analytical/research.
  </Accordion>
  <Accordion title="Per-template Naming">
    A design principle: when a section's cognitive mode differs across templates, use a different section name per template. When the mode is the same, use a unified name. Applied to visual sections (per-template: Value Flow, Diagrams, Findings Map) vs knowledge capture sections (unified: Fact Candidates, Strategic Insights). Decision criterion: "Does the cognitive mode CHANGE between templates?"
  </Accordion>
</AccordionGroup>

## Task concepts

<AccordionGroup>
  <Accordion title="Status Flow">
    The ordered sequence of task statuses from TODO to DONE. Full diagram and transitions are in [conventions §5](/reference/conventions).

    ```
    ⬜ TODO → 📝 HL_DRAFT → 🔬 RES → 🟡 TS_DRAFT → 🟠 ONB → (develop) → 🟢 RF → 🔍 REV → 📚 KNW → ✅ DONE
    ```

    9 named statuses: TODO, HL_DRAFT, RES, TS_DRAFT, ONB, RF, REV, KNW, DONE (plus BLOCKED). RES and KNW are optional.
  </Accordion>
  <Accordion title="KNW (Knowledge Capture)">
    The post-review status indicating that the tfw-docs and tfw-knowledge workflows have been applied. Triggered after a REVIEW APPROVE verdict. REVIEW §4 contains two markers: `tfw-docs: Applied/N/A` and `tfw-knowledge: Applied/N/A`. Both markers set → status transitions to DONE. For trivial tasks, the reviewer can pre-mark both as N/A during review.
  </Accordion>
  <Accordion title="Phase">
    A bounded unit of work within a multi-phase task. Each phase has its own HL → TS → ONB → RF → REVIEW cycle. Phases are named with letters (A, B, C) or numbers. All phases are subject to scope budgets.
  </Accordion>
  <Accordion title="Scope Budget">
    Limits per phase calibrated for AI executor agents. Exceeding limits degrades quality — when exceeded, split the phase. Values are configured in `tfw.scope_budgets` in `PROJECT_CONFIG.yaml`. Defaults: 14 files per phase, 8 new files, 1200 LOC, 12 modified files.
  </Accordion>
  <Accordion title="Task Board">
    A Markdown table in `README.md` — the single source of truth for task statuses. Updated by every TFW workflow. The ID column must be a relative link to the task folder: `[PROJ-1](tasks/PROJ-1__title/)`.
  </Accordion>
  <Accordion title="Task Naming">
    Format: `{PREFIX}-{N}__{short-title}`. Full naming rules and file conventions are in [conventions §4](/reference/conventions).
  </Accordion>
</AccordionGroup>

## Research concepts

<AccordionGroup>
  <Accordion title="Stage (Research)">
    One thematic block within RESEARCH: Gather, Extract, or Challenge. Each stage ends with a checkpoint. Stages form a checklist — the agent must cover all three, but the order is flexible.
  </Accordion>
  <Accordion title="Pass (Research)">
    A full round-trip across all three RESEARCH stages (Gather, Extract, Challenge). Minimum 1 pass required. Additional passes cover stages that need deeper investigation. Recommended maximum: 3 passes.
  </Accordion>
  <Accordion title="Iteration (Research)">
    One full round of `/tfw-research` within a multi-iteration task. Each iteration has its own subfolder (`researchN/`), its own RES file (`RES__iterN__*.md`), and a mandatory Iteration Status block. Iteration 1 is standard research. Iteration 2+ builds on predecessor findings and addresses open threads and gaps. Minimum iterations are configurable via `tfw.research.min_iterations` in `PROJECT_CONFIG.yaml` (default: 2).
  </Accordion>
  <Accordion title="iterations.yaml">
    Control file at the task root for multi-iteration research. Created by the Coordinator in `plan.md` Step 6b. Contains: `task_id`, `title`, `min_iterations`, `max_iterations`, and an `iterations` array tracking each iteration's number, focus, hypotheses, status, and RES file path. The Coordinator owns this file — researchers read it, the Coordinator updates it after each iteration.
  </Accordion>
  <Accordion title="min_iterations">
    A configurable hard floor for research iterations. Default: 2 (from `tfw.research.min_iterations` in `PROJECT_CONFIG.yaml`). The Coordinator gate in `plan.md` Step 6c blocks TS until this many iterations are complete. The Coordinator can override this per task in `iterations.yaml`. Rationale: researchers optimize for speed; structural enforcement ensures minimum depth.
  </Accordion>
</AccordionGroup>

## Roles

For the full roles breakdown, see [roles](/concepts/roles).

<AccordionGroup>
  <Accordion title="User (Human)">
    Approves HL and TS before execution. Provides secrets via environment variables. Reviews RF outputs. Final authority on task closure.
  </Accordion>
  <Accordion title="Coordinator (AI)">
    Writes HL and TS. Manages the Task Board. Hands off to Researcher, Executor, and Reviewer. Runs: plan, resume, docs, knowledge, release, update, config, init workflows.
  </Accordion>
  <Accordion title="Researcher (AI)">
    Dedicated research agent. Writes RES and stage files in the `research/` subfolder. Follows the OODA loop per stage. Hard Stop: after writing RES, says "Research complete. Continue with `/tfw-plan`."
  </Accordion>
  <Accordion title="Executor (AI)">
    Reads the approved TS. Writes ONB before starting. Implements changes. Makes incremental commits. Writes RF documenting results. Reports observations (tech debt, issues).
  </Accordion>
  <Accordion title="Reviewer (AI — coordinator in review mode)">
    Reads RF and TS (for DoD verification). Writes REVIEW file with 9-point checklist. Triages executor Observations to TECH_DEBT.md. Cannot: write code, write ONB, write RF, or modify HL/TS.
  </Accordion>
</AccordionGroup>

## Infrastructure

<AccordionGroup>
  <Accordion title="Topic File">
    A per-category knowledge file in the `knowledge/` folder at the project root. Contains verified facts in a structured table. Template: `.tfw/templates/TOPIC_FILE.md`. Updated by `/tfw-knowledge` consolidation.
  </Accordion>
  <Accordion title="Knowledge Gate">
    A periodic consolidation checkpoint in Phase 0 of `plan.md`. Mode is configurable: `hard` (stop + justification required), `soft` (reminder only), `off` (skip). Controlled by `tfw.knowledge.gate_mode` in `PROJECT_CONFIG.yaml`.
  </Accordion>
  <Accordion title="Consolidation">
    The 4-phase process for converting Fact Candidates into verified project knowledge: Orient → Gather → Consolidate → Prune. Executed via the `/tfw-knowledge` workflow.
  </Accordion>
  <Accordion title="Config Sync Registry">
    A table in the `config.md` workflow that maps `PROJECT_CONFIG.yaml` keys to their inline display locations. The AI agent reads the registry to find where values appear, compares with the YAML, and proposes updates.
  </Accordion>
  <Accordion title="Tool Adapter">
    A tool-specific entry point (CLAUDE.md, .cursor/rules, .agent/workflows/) that references `.tfw/` as the single source of truth. See [adapters overview](/adapters/overview) and [conventions §9](/reference/conventions).
  </Accordion>
  <Accordion title="PROJECT_CONFIG.yaml">
    Per-project configuration file in `.tfw/`. Defines: project name, task prefix, execution engine, template paths, workflow paths, scope budgets, research limits, knowledge settings, and task statuses. Full reference: [PROJECT_CONFIG.yaml](/reference/project-config).
  </Accordion>
  <Accordion title="VERSION">
    Single-line file in `.tfw/` containing the current framework version in semver format (MAJOR.MINOR.PATCH). Current version: `0.8.2`.
  </Accordion>
  <Accordion title="CHANGELOG.md">
    Structured version history in `.tfw/`. Follows Keep a Changelog format. Full history: [changelog](/reference/changelog).
  </Accordion>
  <Accordion title="Compilable Contract">
    Build-time specification for deterministic compilation of TFW artifacts into documentation. Defines the Source Manifest, Reference Format, and Output Structure. Located at `.tfw/compilable_contract.md`.
  </Accordion>
  <Accordion title="Multi-phase Handoff">
    A convention for tasks with 3 or more phases: the master HL §4 includes a Context block per phase (Requires, Shared files, Key decisions, Deliverables), enabling independent coordinators to write a Phase HL without reading all research.
  </Accordion>
  <Accordion title="RESEARCH (pipeline stage)">
    The stage between HL and TS in the planning pipeline. Structured investigation: gathering information, extracting hidden knowledge, critical analysis. Can also run standalone via `/tfw-research`. Always produces a RES artifact.
  </Accordion>
</AccordionGroup>

## Concept taxonomy

| Concept | Definition | Where it lives |
|---|---|---|
| **Document Type** | Type of artifact: HL, RES, TS, ONB, RF, REVIEW | This glossary — Artifact Types |
| **Template** | Canonical format for a document type | `.tfw/templates/` |
| **Workflow** | Tool-agnostic process description (plan, research, handoff...) | `.tfw/workflows/` |
| **Adapter Command** | Tool-specific invocation of a workflow (slash command, skill) | `.claude/commands/`, `.agent/workflows/` |
| **Status** | Process status of a task on the board | `PROJECT_CONFIG.yaml` `tfw.statuses` |
